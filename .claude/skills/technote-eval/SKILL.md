# TechNote 품질 평가 실행 · 해석

`backend/evals/run_eval.py`는 실제 매뉴얼로 질문을 만들어 파이프라인에 흘리고
routing / document / pages 축으로 자동 채점합니다. 실행 자체는 명령 한 줄이지만,
**축마다 신뢰도가 달라서 해석에서 대부분의 실수가 납니다.**

## 가장 중요한 것: 전체 통과율은 품질 지표가 아닙니다

케이스 PASS는 모든 축을 AND로 묶기 때문에, 노이즈가 큰 `pages`가 전체 통과율을 지배합니다.
실측 기준선 (매일 06:00 `--generate 20` 루틴, 2026-07-15 ~ 07-21):

| 실행일 | routing | document | pages | 전체 |
|---|---|---|---|---|
| 07-15 | 20/20 | 20/20 | 14/20 | **14/20** |
| 07-16 | 20/20 | 20/20 | 11/20 | **11/20** |
| 07-17 | 20/20 | 20/20 | 12/20 | **12/20** |
| 07-19 | 20/20 | 20/20 | 18/20 | **18/20** |
| 07-20 | 20/20 | 20/20 | 16/20 | **16/20** |
| 07-21 | 20/20 | 19/20 | 13/20 | **13/20** |

전체 통과율이 11/20에서 18/20까지 출렁이지만 **routing·document는 거의 매회 만점**입니다.
변동은 전부 `pages` 때문입니다. 그러니 "11/20 통과, 품질 저하" 같은 보고는 사실상 오보입니다.
**routing과 document를 먼저 보고, 전체 통과율은 마지막에 참고로만 언급하세요.**

## 실행 모드 고르기

작업 디렉토리는 `backend/`이고, venv 활성화 없이 `venv/bin/python`을 직접 쓰면 됩니다.

| 목적 | 명령 |
|---|---|
| 배포 백엔드 품질 점검 (기본) | `venv/bin/python -m evals.run_eval --generate 20` |
| 코드 수정 검증 (서버·배포 없이) | `venv/bin/python -m evals.run_eval --local` |
| 빠른 스모크 | `venv/bin/python -m evals.run_eval --only greeting` |
| 답변 내용까지 채점 | `venv/bin/python -m evals.run_eval --judge` |
| 대규모 고정 벤치마크 (500문항) | `venv/bin/python -m evals.run_eval --dataset evals/claude_dataset.yaml --equiv` |
| 로컬 서버 대상 | `--base-url http://localhost:8000` 추가 |

고를 때 판단 기준:

- **`--generate N`** — 매 실행 새 질문을 합성합니다. 회귀 탐지에는 좋지만 질문이 매번 달라
  케이스 단위 비교는 불가능합니다. `keywords`/`judge`는 미리 써둔 답이 없어 건너뜁니다.
- **`--judge`는 `dataset.yaml`에서만 의미가 있습니다.** `--generate`와 같이 주면 조용히 무시됩니다.
- **`--local`** 은 HTTP 없이 파이프라인을 프로세스 내에서 직접 호출합니다. 배포 전 코드 검증용.
  단 로컬 `.env`(`USE_LOCAL_STORAGE=True`)를 따르므로 보는 문서 셋이 프로덕션과 다릅니다.
- **`--min-pass`는 웬만하면 주지 마세요.** 통과율이 pages 노이즈에 지배되므로 게이트로 쓰면
  멀쩡한 실행이 exit 1로 떨어집니다. CI 게이트가 꼭 필요할 때만 낮게 잡으세요.

### 실행 시간 — 기본 타임아웃으로는 반드시 죽습니다

케이스가 순차 실행이라 `--generate 20`은 **대략 5~6분**(케이스당 12초 + 질문 합성) 걸립니다.
Bash 기본 타임아웃 2분으로는 중간에 잘리므로 `timeout: 600000`을 명시하거나 백그라운드로 돌리세요.
`claude_dataset.yaml` 500문항은 **수 시간**이라 반드시 백그라운드로 실행합니다.

결과는 콘솔 요약과 함께 `backend/evals/results/<타임스탬프>.json`에 저장됩니다.

## 결과 해석

### routing — 신뢰도 높음, 실패하면 진짜 문제
일상대화 / 기술질문 / 되묻기 분기가 기대와 맞는지. 평소 만점이라 1건만 틀려도 짚을 가치가 있습니다.

단, detail에 `실제 error`라고 찍힌 건 **라우팅 판단이 틀린 게 아니라 케이스가 죽은 것**입니다
(HTTP 401, TIMEOUT 등). 실제로 2026-07-13 실행에서 토큰 만료로 102건이 무더기 `error`가 됐고,
이걸 라우팅 실패로 읽으면 완전히 잘못된 결론이 납니다. 실행 오류는 채점 결과가 아니라
환경 문제로 분리해서 보고하세요.

### document — 신뢰도 높음, 남은 실패는 대개 진짜
`--generate`에는 문서 동등성 판정이 켜져 있어 중복 판본이나 모델 비종속 질문으로 인한
가짜 실패는 자동 통과 처리됩니다. detail 표기를 읽어 성격을 나누세요:

- `동등 문서 인정` → 가짜 실패가 걸러진 것. 언급할 필요 없습니다.
- `실오답 확인` → 동등성 판정에서도 불인정된 **진짜 오답**. 반드시 보고합니다.
- `되묻기 후보에 소스 누락` → 검색이 정답 문서를 후보로도 못 띄움 = **recall 문제**.
- `되묻기 후보에 소스 포함`인데 실패 → 후보엔 떴는데 자동선택이 틀림 = **precision 문제**.

recall이냐 precision이냐로 고칠 지점이 완전히 달라지므로(전자는 검색·인덱싱, 후자는 선택 LLM),
document 실패를 보고할 때는 이 구분을 반드시 함께 적으세요.

### pages — 참고용, 개별 실패를 나열하지 마세요
ToC 챕터 시작점을 근사 정답으로 쓰기 때문에, 실제 내용이 챕터 뒤쪽에 있으면 그냥 틀립니다.
방대한 매뉴얼일수록 오차가 큽니다. **통과율만 한 줄 언급**하고 개별 케이스를 파고들지 마세요.
사용자가 pages를 콕 집어 묻는 게 아니라면 원인 분석에 시간을 쓸 가치가 없습니다.

### 문체별 성적
생성 문항의 30%는 전보식(terse — "2051 서보 알람"처럼 키워드만), 70%는 문장형입니다.
terse의 document 통과율이 문장형보다 눈에 띄게 낮으면 "키워드 빈약한 실사용 입력에서
문서선택이 약하다"는 신호이니 따로 짚어주세요. 실제로 실패는 terse에 몰리는 경향이 있습니다.

## 이력 비교 (회귀 확인)

`scripts/compare_results.py`가 결과 JSON들을 읽어 축별 추이와 실패 성격을 정리해줍니다.
직접 JSON을 파싱하지 말고 이걸 쓰세요 — 위의 해석 규칙(실행 오류 분리, recall/precision 분류,
pages 노이즈 경고)이 이미 들어 있습니다.

```bash
# backend/ 에서 실행
python3 ../.claude/skills/technote-eval/scripts/compare_results.py --dataset generated --last 7
python3 ../.claude/skills/technote-eval/scripts/compare_results.py --diff        # 최신 vs 직전
python3 ../.claude/skills/technote-eval/scripts/compare_results.py --failures    # 최신 실행 실패 상세
```

**생성 데이터셋은 케이스 단위 비교가 성립하지 않습니다** — 질문이 매번 새로 합성되니까요.
"어제 통과했는데 오늘 깨진 질문"을 찾으려 하지 말고 축별 비율로 보세요. 고정 데이터셋
(`dataset.yaml`, `claude_dataset.yaml`)만 케이스 단위 diff가 의미 있고, 스크립트가 알아서 구분합니다.

단발성 변동에 과잉 반응하지 마세요. 20문항 표본이라 document 1건 차이는 5%p로 크게 보이지만
표본 노이즈일 수 있습니다. 여러 회차 추이를 함께 보고 판단하세요.

## 보고 형식

실행 후 이 순서로 정리하면 사용자가 바로 판단할 수 있습니다:

```
## 핵심 지표
- routing: N/20   (실패 시 질문 → 기대 vs 실제)
- document: N/20  (실패 시 질문 → 기대 문서 vs 선택 문서, recall/precision 구분)

## 참고
- pages: N/20 (근사 지표)
- 문체별 document: terse N/M, sentence N/M
- 평균 응답 N초 (최대 N초)
- 전체 통과율 N/20 — pages 영향 포함

## 판단
(이전 회차 대비 회귀 여부 한두 줄)
```

routing·document가 만점에 가까우면 길게 쓸 필요 없습니다. 두세 줄로 끝내고,
실패가 있을 때만 그 축을 파고드세요.

## 오류 처리

- **대부분 HTTP 401** → `backend/evals/.env`에 `EVAL_JWT_SECRET`(프로덕션 JWT_SECRET)이 필요합니다.
  `backend/.env`에 넣으면 안 됩니다 — pydantic Settings가 선언되지 않은 키를 거부해서
  앱 부팅 자체가 `extra_forbidden`으로 실패합니다.
- **후반 케이스만 무더기 401** → 토큰 수명(24시간)을 넘긴 장시간 실행입니다.
- **TIMEOUT 다수** → Cloud Run 콜드스타트이거나 배포 백엔드 이상. 재실행 전에 헬스체크부터 하세요.
- **"생성할 ToC 항목이 없습니다"** → 배포 백엔드 `GET /documents` 응답이나 문서 상태를 확인합니다.

## 범위

기본은 **실행하고 해석해서 보고하는 것까지**입니다. 실패 원인이 명확해 보여도 사용자가
요청하기 전에 소스를 고치지 마세요 — 표본 노이즈일 수 있고, 어느 계층(검색 vs 선택 LLM vs
프롬프트)을 손댈지는 사용자가 결정할 문제입니다. 원인 가설과 근거까지 제시하고 멈추세요.

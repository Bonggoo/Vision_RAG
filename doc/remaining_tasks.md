#### 1. ToC 검색 아키텍처 개선 + 되묻기 로직

> **현재 문제점 3가지:**
> 1. 모든 문서의 ToC를 한 프롬프트에 몰아넣음 → 토큰 낭비 (~40,000 토큰/요청)
> 2. ToC가 3000자 초과 시 30개 항목으로 잘림 → 중요 항목 누락
> 3. 여러 문서에 같은 코드(E1002 등) 존재 시 혼동 → 엉뚱한 문서 선택

##### 📐 현재 구조 vs 개선 구조

```
[현재 — 1회 LLM 호출]
  50개 문서 (메타+ToC 전부) → LLM → 문서+페이지 한번에 선택
  ❌ ToC 잘림, ❌ 토큰 40,000개, ❌ 문서 혼동

[개선 — 2단계 분리 + 되묻기]
  1단계: 메타데이터만 (제목, 제조사, 모델) → LLM → 문서 선택 (~2,500 토큰)
    ├─ 확신 높음 → 2단계 진행
    └─ 확신 낮음 → 사용자에게 선택지 제시 (되묻기)
  2단계: 선택된 문서의 ToC 전체 (잘림 없음) → LLM → 페이지 선택 (~1,500 토큰)
```

##### ⚡ 성능 비교

| | 현재 (1회 호출) | 개선 (2회 호출) |
|---|---|---|
| 총 입력 토큰 | ~40,000 | **~4,000** |
| 예상 시간 | 2-3초 | **~1초** |
| API 비용 | 높음 | **1/10** |
| ToC 잘림 | 30개 항목만 | **전체 검색** |
| 정확도 | 잘린 ToC로 추론 | **전체 ToC로 추론** |

##### 🔄 되묻기 (확신도 기반 분기)

```
"미쓰비시 MR-J5 E1002 알람 원인"
  → LLM: 확신 높음 (문서 명확) → 바로 답변

"E1002"
  → LLM: 확신 낮음 (여러 문서에 존재)
  → "E1002 알람이 여러 매뉴얼에 있습니다. 어떤 장비인가요?
     • 미쓰비시 MR-J5 서보드라이브
     • 야스카와 Σ-V 서보드라이브
     • 파나소닉 MINAS A6 서보드라이브"
  → 사용자 선택 후 답변
```

**되묻기 조건:**

| 조건 | 행동 |
|---|---|
| 문서 1개에 확실히 매칭 | 바로 답변 |
| 비슷한 점수로 여러 문서 매칭 | 선택지 제시 |
| 관련 문서 없음 | "관련 내용을 찾지 못했습니다" |
| 대화 맥락에 문서 힌트 있음 | 맥락 활용해서 바로 답변 |

##### 🗺️ 향후 확장 로드맵 (문서 200개+ 시)

> 문서가 200개를 넘어가면 1단계(메타데이터 LLM 선택)도 느려짐.
> 이때 1단계만 **ToC 임베딩 벡터 검색**으로 교체.

```
[현재 50개]  → 2단계 분리 + 되묻기 (이번 작업)
[200개 넘으면] → 1단계를 임베딩 RAG로 교체

  구체적 구현:
  - 문서 업로드 시 ToC 항목마다 컨텍스트 보강 텍스트 생성
    예: "[미쓰비시 MR-J5 매뉴얼] > 3장 알람코드 > 3.2 E1002 과전류 알람 (p.45)"
  - Gemini text-embedding-005 로 임베딩 → GCS에 벡터 저장
  - 질문 시 벡터 검색 (0.1초) → top-10 후보 → 되묻기 판단
```

**일반 RAG와 다른 점 (Vision RAG 특성):**

| | 일반 RAG | Vision RAG (우리) |
|---|---|---|
| 임베딩 대상 | 문서 텍스트 청크 | **ToC + 메타데이터** |
| 검색 목적 | 답변에 쓸 텍스트 찾기 | **볼 페이지 번호 찾기** |
| 답변 생성 | 검색된 텍스트 기반 | **페이지 이미지 Vision** |
| 정확도 영향 | 직접적 (답변 품질 좌우) | 간접적 (페이지만 맞으면 됨) |

> ToC RAG가 틀려도 최악은 "엉뚱한 페이지를 봄"이지, "잘못된 답변"이 아님.
> Vision이 실제 페이지를 보고 판단하기 때문.

#### 5. 참조 페이지 PDF 보기 방식 ⏸️ 보류

> 방향 결정 필요: 모달 이미지 뷰어 vs 실제 PDF 열기 + 해당 페이지 이동

**검토 중인 2가지:**

| 방향 | 장점 | 단점 |
|:---|:---|:---|
| A. 모달 이미지 뷰어 | 구현 간단 | 이미지라 텍스트 선택 불가 |
| B. PDF 직접 열기 (`#page=N`) | 원본 품질, 텍스트 검색 가능 | PDF 뷰어 라이브러리 필요 |

> 방향 B의 경우: 백엔드에 `download_document_url()` (GCS Signed URL)이 이미 있으므로 `URL#page=45` 형태로 바로 이동 가능

**예상 공수**: 방향 A → 1일 / 방향 B → 2~3일

# 근접 중복 문서 처리 설계 (Near-Duplicate Document Handling)

- 상태: **채택 (L1 구현 완료)** · 2026-07-13
- 관련 코드: `backend/app/services/dedup_service.py`, `backend/app/routers/upload.py`, `backend/app/services/pdf_service.py`

## 1. 배경 / 문제

지금은 관리자(개발자) 1인이 쓰는 단계라, 유사·중복 문서가 생기면 **품질 평가
루틴(`evals`)으로 찾아 수동 삭제**하면 됐다. 하지만 곧 다중 사용자 서비스가 되면
각 사용자가 같은 매뉴얼을 여러 번, 혹은 여러 사용자가 같은 공개 매뉴얼을 각자
업로드하게 된다. "평가해서 지운다"는 1인 관리자용 임시방편이고, 다중 사용자에선
중복을 **업로드 시점의 제품 기능**으로 다뤄야 한다.

현재 방어선은 업로드 preflight의 **SHA-256 바이트 해시** 중복검사뿐이라, 완전히
동일한 파일만 잡는다. 같은 매뉴얼의 **재추출본**(페이지가 1~4장 다른 PDF 등)은
바이트가 달라 통과해버린다. 실제로 `키엔스 CV-X 시리즈 사용자 매뉴얼`이 페이지
수만 미세하게 다른 채(1278p / 1274p / 1274p) **3벌** 쌓여 있었고, 이는 문서 선택
시 후보를 쪼개 검색 품질을 떨어뜨리는 실측 원인이 되었다.

### 문제를 두 갈래로 분리

| | 문제 | 성격 | 영향 |
|---|---|---|---|
| **P1** | 한 사용자가 같은 매뉴얼을 여러 번 업로드 | 검색 품질 | 후보 분산 → 오선택 |
| **P2** | 여러 사용자가 같은 공개 매뉴얼을 각자 업로드 | 비용 | 저장·전처리 중복 |

**핵심 관찰:** 문서는 이미 사용자별로 격리되어 있다(GCS 경로 `users/{owner_email}/...`,
검색도 `get_all_documents_async(owner_email=...)`). 따라서 A 사용자의 중복이 B의
검색을 오염시키지 않는다 — **P1은 각 사용자 코퍼스 내부 문제로 국한**되고, P2는
검색 피해 없는 순수 비용 문제다. 이 격리 덕분에 문제 범위가 크게 줄어든다.

## 2. 결정

**L1(사용자 내부 근접 중복 감지)을 지금 구현한다.** L2/L3는 미래 옵션으로 남긴다.

### 왜 감지를 preflight가 아니라 분석 파이프라인 끝에서 하나

콘텐츠 기반 유사도를 재려면 ToC·제조사·모델 등 **추출된 콘텐츠**가 필요하다.
그런데 preflight 시점엔 PDF가 아직 GCS에 올라가지도, 파싱되지도 않았다(가진 건
파일명·크기·바이트해시뿐). 따라서 콘텐츠 지문 비교는 ToC/모델 추출이 끝나는
**분석 파이프라인 마지막 단계**에서 수행한다. 결과는 새 문서 메타데이터의
`similar_documents` 필드에 실려 문서 목록 API로 그대로 흘러간다.

### 왜 자동 삭제/차단이 아니라 '감지 후 사용자 선택'인가

근접 중복 자동 삭제는 위험하다. 코퍼스의 `MELSERVO-J4 트러블슈팅편(58p)` vs
`MELSERVO-J4 전체(618p)`는 파일명이 비슷하지만 **다른 문서**다 — 자동으로 지웠으면
오답이 된다. 그래서 **완전 동일 콘텐츠(바이트 해시)만 차단(409)**하고, 그 외
근접 중복은 **감지해서 노출만** 한다. 사용자가 '유지 / 교체 / 삭제'를 결정한다.

## 3. L1 설계 상세

### 콘텐츠 지문 = ToC 제목 토큰 집합

두 문서의 유사도를 **추출된 ToC 제목들의 토큰 집합 Jaccard**로 잰다.
- 같은 매뉴얼의 재추출본은 ToC가 거의 동일 → 높은 유사도.
- 부수적 장점: ToC 제목 비교는 **제조사 표기 차이에 무관**하다
  (`미쓰비시`/`미쓰비시전기`/`미쯔비시`가 섞여 있어도 ToC 내용이 같으면 잡힌다).
- ToC가 없는 스캔 문서는 파일명 토큰으로 대체하고, 보조로 `model_series` 일치 +
  페이지 수 근접 규칙을 쓴다. **단 이 보조 규칙은 둘 중 하나라도 ToC가 없을
  때만** 적용한다 — 둘 다 ToC가 있는데 제목이 다르면 같은 제품의 '다른 문서'
  (카탈로그 vs 설정가이드)일 수 있어 오탐이 되기 때문이다.

### 임계값 = 0.7 (실측 코퍼스 55개로 교정)

실제 보유 매뉴얼 55개로 모든 쌍의 ToC Jaccard를 계산해 교정했다:

| 쌍 | 유사도 | 판정 |
|---|---|---|
| CV-X 사용자 매뉴얼 ×3 (재추출본) | 0.989 ~ 0.997 | 🔴 중복 (정탐) |
| QD77MS 심플모션 (`미쓰비시`/`미쓰비시전기`) | 1.000 | 🔴 중복 (정탐) |
| 시리얼 통신 / 커뮤니케이션 (다른 번역본) | 0.09 | 미검출 |
| 시리얼 기본편 / 응용편 (별권) | < 0.35 | ⚪ 정상 (오탐 아님) |
| CZ-V20 카탈로그 / 설정가이드 | 0.13 | ⚪ 정상 (오탐 아님) |

진짜 중복은 0.99+, 정당한 별권은 0.35 미만으로 명확히 갈려 **임계값 0.7에서
오탐 0건**으로 4개 중복 쌍(CV-X 3 + QD77MS 1)만 정확히 잡힌다. 번역본이 다른
`시리얼 통신/커뮤니케이션`은 미검출되지만, 이는 보수적으로 둔 결정이다 — 다른
번역본은 둘 다 유효할 수 있어 굳이 중복 경고를 띄우지 않는 편이 안전하다(평가
하니스의 문서 동등성 판정이 채점 단계에서 별도로 이 케이스를 흡수한다).

### 재사용 시너지

`dedup_service.similarity`가 재는 "두 문서가 같은가"는 평가 하니스의
`_judge_document_equivalence`("선택된 문서로도 답할 수 있는가")와 같은 계열의
문제다. eval용으로 만든 판정 감각이 업로드 dedup 제품 기능으로 이어진다.

## 4. 남은 작업 / 미래 옵션

- **프론트엔드(L1 마무리):** `similar_documents`가 있는 문서에 "기존 'X'와 매우
  유사합니다 — 유지 / 교체 / 삭제" 배너를 노출. 백엔드 감지·API는 완료됨.
- **L2 (검색 중복 내성, 보류):** 되묻기 후보에 거의 동일한 문서가 둘 이상 뜨면
  하나로 접기. L1이 중복 축적을 애초에 막으면 이 상황이 거의 안 생겨 **구현 보류**.
- **L3 (교차 사용자 dedup / P2, 규모 확장 시):**
  - **콘텐츠 주소 저장**: PDF를 content-hash 기준 한 벌만 저장하고 사용자별
    레코드는 참조만. 같은 파일을 올리면 업로드·전처리를 통째로 건너뜀
    (저장·컴퓨트 절감). 프라이버시: 해시로만 공유, "누가 뭘 올렸는지" 비노출.
  - **공용 매뉴얼 라이브러리**: 공개 카탈로그류는 관리자 큐레이션 공용 라이브러리로
    두고 사용자는 사유 문서만 업로드 → 중복 자체를 감소.

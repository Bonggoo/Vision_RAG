### 촬영/제작 가이드 (macOS)

1. **화면 녹화**: `⌘⇧5` → 영역 선택 녹화. 브라우저 창을 1280×800으로 고정 (일관된 프레임)
2. **연출 준비**: 시연용 계정에 문서 3~4개만 남겨 사이드바를 깔끔하게. 질문은 미리 복사해두고 붙여넣기(오타 리스크 제거)
3. **모바일 장면**: iPhone은 QuickTime 유선 미러링 녹화가 가장 깨끗함
4. **편집**: iMovie(무료)로 컷 편집 + 자막. 자막은 영상 안에 구움(자동재생 GIF는 소리 없음)
5. **GIF 변환** (README용, 10MB 이하 목표):
   ```bash
   ffmpeg -i demo.mp4 -vf "fps=12,scale=960:-1" -c:v gif demo.gif
   # 또는 고품질: brew install gifski → gifski --fps 12 --width 960 -o demo.gif demo.mp4
   ```
6. README 반영: 상단 타이틀 바로 아래 `![demo](doc/assets/demo.gif)`

## 2. ✍️ 기술 블로그

### 글 1 (대표작): "RAG인데 벡터 DB가 없습니다 — 사람이 매뉴얼 찾듯 검색하는 에이전트"

> 타깃: RAG 만들어본/만들려는 개발자. 후킹 포인트: "RAG = 임베딩 + 벡터DB"라는 공식에 대한 반례

**아웃라인:**
1. **문제**: 산업용 장비 매뉴얼 — 수백~1000+ 페이지, 표·회로도·배선도 투성이
2. **일반 해법의 함정**: 청킹하면 표가 찢어지고, 회로도는 텍스트 임베딩이 안 됨. 임베딩 파이프라인 유지비용
3. **관찰**: 숙련 기술자는 매뉴얼을 이렇게 찾는다 — 표지 보고 문서 고름 → 목차 훑음 → 페이지로 점프 → 그 페이지를 '본다'
4. **설계**: 이걸 그대로 에이전트로 — 3단계 파이프라인 (문서선택 → ToC/텍스트 탐색 → Vision 분석)
   - 아키텍처 다이어그램 (README 것 재활용)
   - 확신이 없으면 되묻기(Clarification) — "아는 척하지 않는 AI"
5. **좋았던 점**: 업로드 즉시 사용(인덱싱 대기 0), 인프라 단순(GCS만), 근거 페이지 제시 가능
6. **한계 (솔직하게)**: 문서 수백 개 규모에선 문서선택 단계가 병목 → 그때는 ToC 임베딩 하이브리드 계획. 여러 매뉴얼에 걸친 공통 명령어(RECV류)의 모호성
7. **결과**: 독립 500문항 벤치마크 93%+ ← "이 수치를 어떻게 믿냐고요? 그게 다음 글입니다" (글 2 예고)
8. 데모 영상 + 라이브 링크(※ 아래 '공개 전 체크' 참고)

### 글 2: "내 RAG가 좋아지고 있는지 어떻게 아는가 — 1인 개발의 LLM 평가 하네스"

> 타깃: LLM 앱 만드는 개발자. 평가(evals)는 요즘 가장 뜨거운 주제인데 실전 사례 글이 드묾

**아웃라인:**
1. "고쳤더니 다른 게 망가졌다" — 눈으로 테스트하던 시절의 공포
2. 골든 데이터셋 + SSE 파싱 자동 채점 (routing/document/pages/keywords/judge 5축)
3. 매번 다른 문제 내기: ToC에서 문항 자동 생성 + 전보식 30% 혼합 ("2051 서보 알람")
4. 가짜 실패 거르기: LLM 동등성 판정 (중복 문서·일반 질문)
5. 출제자와 응시자가 같으면 안 되지: Claude가 낸 독립 500문항
6. 실전 사례: 실패 13건 → 원인 분석 → ToC 증거 주입 → 11건 수정, 회귀 0
7. 매일 자동 실행 루틴 — 품질 회귀를 아침에 알게 되는 삶
